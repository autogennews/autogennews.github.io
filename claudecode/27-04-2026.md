# Claude Code Weekly — April 21–27, 2026

## 🚀 What Actually Mattered

### 1. Anthropic Publishes Postmortem on Weeks of Claude Code Quality Degradation

**What it does:** Anthropic traced weeks of user-reported quality collapse to **three separate engineering mistakes** stacked on top of each other, creating broad, inconsistent degradation that was hard to diagnose:

| Issue | Introduced | Fixed | What Went Wrong |
|---|---|---|---|
| Reasoning effort default lowered to `medium` | Mar 4 | Apr 7 | Latency improved but intelligence noticeably dropped. Users preferred smart-and-slow. |
| Session thinking cache bug | Mar 26 | Apr 10 | Intended to clear stale reasoning from idle sessions once; instead cleared it **every turn**, making Claude forgetful and burning usage limits. |
| Verbosity-reduction system prompt | Apr 16 | Apr 20 | Capped responses to ≤25 words between tool calls. Crushed coding quality. |

All three were reverted by v2.1.116 (Apr 20). Anthropic reset usage limits for all subscribers on April 23.

**Why it matters:** This is the most transparent postmortem Anthropic has published about Claude Code. It explains why many developers felt Claude had been "lobotomized" for weeks — it wasn't one thing, it was three overlapping issues hitting different user slices at different times. The API was never affected; this was entirely a Claude Code harness problem.

Boris Cherny (Claude Code team lead) responded on Hacker News: *"We agree, and will be spending the next few weeks increasing our investment in polish, quality, and reliability."*

**Verification:** ✅ Official — [anthropic.com/engineering/april-23-postmortem](https://www.anthropic.com/engineering/april-23-postmortem), [HN thread](https://news.ycombinator.com/item?id=47878905), coverage by [The Register](https://www.theregister.com/2026/04/23/anthropic_says_it_has_fixed/), [Fortune](https://fortune.com/2026/04/24/anthropic-engineering-missteps-claude-code-performance-decline-user-backlash/), [Simon Willison](https://simonwillison.net/2026/Apr/24/recent-claude-code-quality-reports/)

### 2. Claude Code v2.1.116–119: A Week of Rapid Releases

Five versions shipped between April 20–25. The headline features that actually matter:

**67% faster `/resume` on large sessions (40MB+)**
No more needing to start fresh on long refactors or multi-day branches. This is a real workflow unblock for heavy users.

**Config persistence (`/config`)**
Settings like `/model`, `/theme`, `/effort`, `/keybindings`, and `/permissions` now save to `~/.claude/settings.json` and survive restarts. Eliminates the daily reconfiguration ritual.

**Vim visual modes (`v`, `V`)**
Character-wise and line-wise visual selection with standard operators (`d`, `y`, `c`). Works with visual feedback.

**Custom themes (`/theme`)**
Built-in themes plus hand-editable JSON in `~/.claude/themes/`. Includes "Auto (match terminal)" for dark/light mode sync.

**MCP tool hooks with `duration_ms`**
`PostToolUse` and `PostToolUseFailure` hooks now include execution timing. Useful for CI/CD monitoring and performance tracking of MCP tool calls.

**Expanded PR support**
`--from-pr` now works with GitLab merge requests, Bitbucket PRs, and GitHub Enterprise URLs.

**Forked subagents**
`CLAUDE_CODE_FORK_SUBAGENT=1` enables forked subagent execution — route cheaper tasks to lighter models.

**Sandbox hardening**
Protection against `rm /` and critical directory deletion even with permissive allowlists. Tighter Bash rules, macOS path safeguards.

**Verification:** ✅ Official/Mixed — [Releasebot](https://releasebot.io/updates/anthropic/claude-code), [ton-technotes weekly recap](https://ton-technotes.com/en/blog/2026-04-25-claude-code-weekly-update-v2119/), [changelogs.directory](https://changelogs.directory/tools/claude-code/releases/2.1.119), [WotAI](https://wotai.co/blog/claude-code-2-1-116)

### 3. Opus 4.7's `xhigh` Effort Level Is Now the Default

**What it does:** A new effort tier between `high` (5,000 thinking tokens) and `max` (20,000 thinking tokens), budgeted at **10,000 thinking tokens**. It's now the default for all Claude Code users on Opus 4.7. Other models default to `high`.

**Why it matters:** This is Anthropic's answer to the March debacle of lowering effort to `medium`. The `xhigh` level represents where "most agentic workloads actually want to live" — enough depth for complex multi-file reasoning without the latency and cost of `max`. Opus 4.7 also uses adaptive thinking, scaling compute dynamically up to your effort ceiling.

**Verification:** ✅ Official — [anthropic.com/engineering/april-23-postmortem](https://www.anthropic.com/engineering/april-23-postmortem), [TheSequence](https://thesequence.substack.com/p/the-sequence-ai-of-the-week-847-everything), [DataCamp benchmark](https://www.datacamp.com/de/tutorial/opus-4-7-project)

### 4. GPT-5.5 Launches — Competitive Pressure Heats Up

OpenAI released GPT-5.5 on April 23, triggering a wave of head-to-head comparisons with Opus 4.7:

| Benchmark | GPT-5.5 | Opus 4.7 | Winner |
|---|---|---|---|
| Terminal-Bench 2.0 | 82.7% | 69.4% | GPT-5.5 (+13.3) |
| SWE-Bench Pro | 58.6% | 64.3% | Opus 4.7 (+5.7) |
| MRCR v2 (long-context retrieval) | 74% | 32.2% | GPT-5.5 |
| Token efficiency | 72% fewer tokens | — | GPT-5.5 |
| Time to first token | ~3s | ~0.5s | Opus 4.7 (6× faster) |

**Community consensus:** GPT-5.5 is more token-efficient and better at long-context retrieval. Opus 4.7 remains stronger for deeply agentic, multi-step coding workflows (the kind Claude Code specializes in) and has dramatically better interactive latency.

**Why it matters for Claude Code users:** If you're using Claude Code for its core strength — autonomous multi-file refactoring, real GitHub issue resolution, long-running agentic tasks — Opus 4.7 still leads. But GPT-5.5's efficiency advantage is real and will pressure Anthropic on cost.

**Verification:** Mixed — [MindStudio comparison](https://www.mindstudio.ai/blog/gpt-55-vs-claude-opus-47-coding-comparison), [LLM Stats](https://llm-stats.com/blog/research/gpt-5-5-vs-claude-opus-4-7), [TowardsAI](https://pub.towardsai.net/gpt-5-5-vs-claude-opus-4-7-benchmarks-trade-offs-and-practical-guidance-for-2026-cebed8574790)

### 5. ⚠️ Known Bugs: Silent Model Swap in v2.1.119/120

**What happened:** v2.1.119 silently routes `claude-opus-4-7` (200K context) to the 1M-context variant. The model picker shows the 200K version selected, but the session actually runs the 1M version. This can cause unexpected cost and behavior differences.

A separate auto-update breakage issue (#53028) also affected some users.

**What to do:** Verify your actual runtime model. Check [GitHub issue #53327](https://github.com/anthropics/claude-code/issues/53327) for status and workarounds. A [community survival checklist](https://gist.github.com/yurukusa/a866b4cd2976486156a00c190c39cef6) covers both bugs.

**Verification:** ✅ Community-confirmed via GitHub issues — [#53327](https://github.com/anthropics/claude-code/issues/53327), [#53028](https://gist.github.com/yurukusa/a866b4cd2976486156a00c190c39cef6)

## 🔧 Useful Tips & Tricks

**Set your effort level once and forget it.** With config persistence in v2.1.119+, run `/effort xhigh` (or whatever you prefer) and it sticks across sessions. No more re-setting every launch. Same for `/theme`, `/model`, `/keybindings`.
*Source: [ton-technotes](https://ton-technotes.com/en/blog/2026-04-25-claude-code-weekly-update-v2119/) — ✅ Official*

**Use MCP hook `duration_ms` for performance monitoring.** The new `PostToolUse` hook includes execution timing, which means you can log slow MCP tool calls in CI or flag performance regressions automatically.
*Source: [Releasebot](https://releasebot.io/updates/anthropic), [claude-howto](https://github.com/luongnv89/claude-howto) — ✅ Official*

**`CLAUDE_CODE_FORK_SUBAGENT=1` for cost control.** Route simpler subtasks to cheaper/faster models while keeping Opus 4.7 on the main thread. Useful for large projects where not every subtask needs max intelligence.
*Source: [ton-technotes](https://ton-technotes.com/en/blog/2026-04-25-claude-code-weekly-update-v2119/) — ✅ Official*

**Use `--from-pr` with GitLab/Bitbucket.** If you're not on GitHub, PR-driven workflows now work across all major platforms. Previously GitHub-only.
*Source: [changelogs.directory](https://changelogs.directory/tools/claude-code/releases/2.1.119) — ✅ Official*

**Comprehensive cheat sheet available.** The community-maintained [Claude Code Cheat Sheet](https://cc.storyfox.cz) covers all keyboard shortcuts, slash commands, CLI flags, MCP servers, memory files, skills, agents, and env vars — updated through v2.1.120.

## 🧭 Bottom Line

**Most important update:** The **April 23 postmortem**. If you felt Claude Code got worse over the past month, you weren't imagining it — three stacked engineering mistakes degraded quality for weeks. All fixed as of v2.1.116. This is the most important thing to know: **Claude Code is back to full quality.**

**Most useful workflow improvement:** **Config persistence** (`/config`). Small feature, big daily impact. Set your effort level, theme, model, and keybindings once. Done.

**Most important model-related change:** **`xhigh` as the new default effort level** for Opus 4.7. It's the right balance for most agentic coding work — noticeably smarter than `high`, without the latency tax of `max`. If you manually set effort before, check whether `xhigh` is now doing what you want.

**Overhyped / low-signal:** The GPT-5.5 vs Opus 4.7 "who wins" discourse. Both are strong. GPT-5.5 is more efficient; Opus 4.7 is better at the deeply agentic work Claude Code is built for. Pick based on your workflow, not benchmarks. Also: Claude Mythos/Claude 5 speculation — none of it affects Claude Code users today.

**What developers should try now:**
1. **Update to v2.1.119+** and verify your model isn't silently swapped (check issue #53327)
2. **Run `/effort xhigh`** if you're on Opus 4.7 — it should already be default, but confirm
3. **Set up config persistence** — `/theme`, `/effort`, `/model` all stick now
4. **Try `/resume` on a large session** — the 67% speedup is real and noticeable on 40MB+ conversations
5. **Explore `CLAUDE_CODE_FORK_SUBAGENT=1`** if you're burning through usage limits on complex projects
